## Verdicts

**Proposal 1 — Closed-shadow-aware `suggested_next_step`**

**Technically sound?** Yes, but has a wiring gap. `_diagnostic_next_step` (stepper.py:59) receives only `instruction`, `libraries`, and `session_id` — it has no access to the DOM snapshot history where `has_possible_closed_shadow_roots` lives. Threading the flag through requires either storing it on the session record (cleanest) or adding the store as a dependency of the helper. Proposal #6 is the prerequisite.

**UX win:** ~3-5 avoided dead-end `dom_selector` calls per closed-shadow encounter. The dbschenker run already hit this loop twice.

**Missed alternative:** Also suppress the `dom_selector` suggestion entirely when the flag is set (not just prepend — it's noise when the root is closed).

**Verdict: LAND** (after #6 lands as infrastructure).

---

**Proposal 2 — Shadow-piercing selector helper**

**Technically sound?** Only for **open** shadow DOM. `>>>` / chained locators won't pierce closed roots — the platform contract that the report itself cites. On dbschenker (13 closed, 1 open), this hint would be correct only 7% of the time and misleading the rest.

**UX win:** Near zero on the actual test data. Would help on selectorshub-class pages with open roots.

**Missed alternative:** Gate the hint on `has_possible_closed_shadow_roots=False`. If closed shadow is suspected, don't suggest piercing — suggest ARIA or page redesign.

**Verdict: REFINE** — conditionally useful, currently misleading on heavy-shadow enterprise pages.

---

**Proposal 3 — Consent-banner pattern library**

**Technically sound?** Gotcha: Usercentrics itself renders behind a shadow root (the failing selector in the run was a shadow-hosted button). A flat CSS pattern list won't reliably match it. OneTrust is light-DOM; Cookiebot varies.

**UX win:** ~5 steps saved on OneTrust pages. Zero on Usercentrics (the dominant enterprise case).

**Missed alternative:** Document it as a pre-flight skill snippet rather than shipping code — patterns drift, and shadow-hosted consent UIs need `>>>` or CDP, not pattern strings.

**Verdict: SKIP** — the problem it targets (Usercentrics) is itself a closed-shadow problem; the proposal doesn't address that root cause.

---

**Proposal 4 — ARIA-derived selector hints**

**Technically sound?** Yes. The ARIA YAML already encodes `textbox "Origin"` — parsing that into `role=textbox[name="Origin"]` is pure string processing in `_aria_summary` (snapshot.py:379). Browser Library's Playwright backend accepts role locators directly.

**UX win:** Eliminates the entire "read ARIA → mentally derive locator → try it" loop. The agent gets ready-to-paste locators in the summary dict, no extra round-trip.

**Missed alternative:** Include the top-5 interactive nodes (textboxes, buttons with name) rather than all 324 — keeps summary size bounded.

**Verdict: LAND** — highest leverage of the six; pure upside on an already-executed snapshot.

---

**Proposal 5 — Wait-for-state in batched setup**

**Technically sound?** Valid concern; `Sleep 5s` is fragile. But adding `wait_for: str | None` to a helper is meaningless unless the caller knows the right light-DOM sentinel — which requires prior inspection. The API design is underspecified.

**UX win:** Real but caller-dependent. Without a known stable selector, the wait just shifts the problem.

**Missed alternative:** Surface this as a documented pattern in the skill template with an example stable selector (e.g. wait on `css=nav` or `css=header`). Zero code, immediate value.

**Verdict: REFINE** — valid as documented guidance; premature as a formal API parameter.

---

**Proposal 6 — Track `closed_shadow_probe` on session metadata**

**Technically sound?** Yes. Write the count to the session record during `_capture_dom` (snapshot.py:322-327) and expose it on `rf_session(action='get')`. Low-risk, isolated change.

**UX win:** Primarily enables #1. Secondarily lets agents adopt ARIA-first before the first failed step.

**Verdict: LAND** (prerequisite for #1, small standalone value).

---

## Ranking I would adopt

1. **#4 — ARIA-derived selector hints** (~20 lines in `_aria_summary`). Highest agent ROI, no new state, no wiring changes. Every ARIA call becomes immediately actionable.

2. **#6 → #1 combined** (~50 lines total). #6 writes the closed-shadow flag to session state; #1 reads it to hard-block the dead-end `dom_selector` loop. Together they close the most wasteful agent anti-pattern observed in the run.

3. **#2 refined** — once #6 is in place, gate the piercing-selector hint on `has_possible_closed_shadow_roots=False`. That makes it correct on open-shadow pages without being noise on closed-shadow ones.
