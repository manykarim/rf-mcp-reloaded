## Verdict on Each Proposal

**1. Closed-shadow-aware `suggested_next_step`** (`stepper.py:_diagnostic_next_step`)
- **Sound?** The mechanism is real — `_diagnostic_next_step` is the exact landing site. The `has_possible_closed_shadow_roots` flag is already computed in the DOM summary. The missing piece is that the stepper doesn't currently *receive* that flag; it would need the session record to carry a `dom_summary` dict (or at minimum a `has_possible_closed_shadow_roots` bool) so the helper can branch on it. That's a thin wire — ~5 lines of extra data flow.
- **UX win?** High. Stops the agent from looping on `dom_selector` for ~13 wasted calls per stuck scenario on dbschenker-class pages. ~30 lines, zero new APIs.
- **Missed alternative:** A simpler version just checks if the failing selector is an attribute selector (`[data-test-id=...]`) and unconditionally warns about shadow piercing. Lower fidelity but zero cross-session state needed.
- **Verdict:** LAND — narrow scope, existing landing site, proven data signal.

**2. Shadow-piercing selector helper**
- **Sound?** `>>>` piercing combinator is real in Playwright/Browser Library. The gotcha: `dom_selector` snapshot uses `Get Property`/`Get Element Attribute` against the *light DOM host* — it cannot validate a selector that reaches into a closed root. So the helper would be telling the agent "try `>>>`" for something the tool itself cannot verify. Plausible but one step removed from being self-consistent.
- **UX win?** Medium. ~2-3 round-trips saved per shadow-stuck step.
- **Missed alternative:** Land proposal 1 first; an agent that already knows there's a closed shadow boundary will naturally reach for `>>>` or ARIA without being told.
- **Verdict:** REFINE — good intent, but defer until proposal 1 lands so the advisory context exists.

**3. Consent-banner pattern library**
- **Sound?** Pattern recognition is trivial (string matching on known class/id names). No hidden mechanism risk.
- **UX win?** Small. Only helps the cookie-banner case, which is one specific failure mode among many. Agents discover the pattern in 1-2 failed steps anyway.
- **Missed alternative:** A single entry in `agent_prompt.txt` listing these 3 selectors costs nothing and ships immediately.
- **Verdict:** SKIP — low leverage, belongs in docs/prompting not code.

**4. ARIA-derived selector hints**
- **Sound?** Technically straightforward — parse `_aria_summary` YAML, emit `role=<role>[name="<name>"]` templates. But the dbschenker data settles the hidden gotcha: 3 textboxes / 16 buttons / 0 comboboxes for what should be a multi-step booking form. The form fields are *inside closed shadow roots*. ARIA cannot see them. On open-shadow pages (selectorshub), this is excellent. On enterprise SPAs, it's partial at best.
- **UX win?** High on open-shadow pages, near-zero on closed-shadow-heavy pages like dbschenker. Average case probably ~50% hit rate across real targets.
- **Missed alternative:** Combine with a `has_possible_closed_shadow_roots` check — only surface ARIA-derived hints when ARIA coverage is likely complete.
- **Verdict:** REFINE — worth doing but needs a guard that gates it on shadow-open pages or degrades gracefully.

**5. Wait-for-state in batched setup**
- **Sound?** `Wait For Elements State` is a real Browser Library keyword. Correct observation that `Sleep 5s` is fragile.
- **UX win?** Small. Setup robustness only; doesn't affect the main agent loop. The 5s sleep is ugly but rarely the breaking point.
- **Missed alternative:** The `wait_for: str | None = None` hint to batched-setup helper is the right API. Keep it minimal.
- **Verdict:** SKIP for now — operational improvement with no strategic value; can revisit when setup fragility becomes a recurring complaint.

**6. Track `closed_shadow_probe.possible_closed_shadow_root_count` on session metadata**
- **Sound?** The probe already runs; the count is already computed. Promoting it to session-level metadata (`rf_session(action='get')`) is purely additive — no mechanism risk, no behavioral break.
- **UX win?** Small-to-medium. Agents can adopt an ARIA-first strategy upfront on heavy-shadow pages, avoiding wasted `dom_selector` calls before the first failure. ~1 call saved per shadow-heavy scenario.
- **Missed alternative:** Proposal 1 achieves the same goal reactively (on failure) without needing a new field. This proposal makes it proactive.
- **Verdict:** REFINE — worth doing but lower priority than 1. Good as a follow-on: the proactive signal complements the reactive advisory.

## Ranking I Would Adopt

1. **#1 — Closed-shadow-aware `suggested_next_step`** (LAND now). 30 lines in `stepper.py`. Stops the infinite `dom_selector` loop on the exact failure mode the dbschenker run exposed. The only prerequisite is surfacing `has_possible_closed_shadow_roots` from the session's latest DOM snapshot — a thin additional field on `SessionRecord`.

2. **#4 — ARIA-derived selector hints** (REFINE then land). When ARIA *can* see the target (open shadow, or light DOM), surfacing `role=textbox[name="Origin"]` in the manifest summary eliminates the re-derivation step entirely. Needs a guard: if `has_possible_closed_shadow_roots=True` and the target role is form-field, suppress the hint and point at proposal 1's advisory instead.

3. **#6 — Session-level `possible_closed_shadow_root_count`** (REFINE then land). Makes the shadow load *proactive* rather than reactive. The agent can decide on ARIA-first before the first failure on pages like dbschenker. Clean additive change; no existing behavior broken.
