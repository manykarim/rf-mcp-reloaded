### What Went Well

- The epic stayed aligned to the architecture boundary instead of expanding MCP into a broad helper surface.
- Each story reinforced the previous one:
  Story 2.1 established the bounded session model, Story 2.2 made context and snapshots attributable, Story 2.3 made fallback repair guidance deterministic, and Story 2.4 proved the flagship repair path end to end.
- Review-driven fixes materially improved the implementation rather than only polishing story files:
  - unsupported transport errors were normalized onto the shared structured path
  - invalid context mutations stopped corrupting session state
  - provider and hint-pack failures became explicit and fail-closed
  - flagship workflow proof moved from static confidence to real executable evidence
- The project now has real packaged proof surfaces:
  the flagship skill ships non-code assets, executable fallback examples, and benchmark JSONL output instead of only architecture narrative.
- Previous Epic 1 guidance to test real invocation paths was applied successfully:
  `uv run`, wheel builds, MCP tool registration, and real Robot execution all caught defects that source-only tests would have missed.

### Challenges and Growth Areas

- The largest recurring defect class was still execution-environment coupling:
  subprocess proof paths, package-install surfaces, and review bundles all exposed assumptions that were invisible in direct source imports.
- External review tooling remained useful but uneven across the epic:
  Claude coverage is well recorded throughout the story files, while kilo and Codex usage appeared later and is less evenly represented in the archived story records.
- The flagship workflow needed multiple hardening passes before it matched the acceptance criteria literally:
  baseline failure proof, deterministic rerun evidence, package inclusion of assets, and idempotence across equivalent Robot syntax were not right on the first pass.

### Root Causes and Patterns

- Real defects clustered around proof claims, not around basic scaffolding:
  the hard part was making “bounded,” “deterministic,” and “portable” true under execution, not describing them in code structure.
- Hidden environment assumptions remain the most likely failure source:
  current working directory, installed tool availability, package-data inclusion, and subprocess output shape all mattered.
- Review passes were highest value when they forced the code through actual runtime seams:
  package builds, CLI invocation, Robot execution, and MCP tool behavior.
- Acceptance-criteria misses tended to happen when success was inferred from nearby evidence rather than proven directly:
  for example, using validation output as a proxy for runnable success or assuming a package asset was shipped because the source file existed.

### Technical Debt, Residual Risks, and Significant Discoveries

- The local machine is still below the declared `uv` baseline, so some bootstrap-enforced behavior remains dependent on a known environment mismatch.
- The flagship workflow now proves its repair path through real Robot execution, but that means proof quality still depends on keeping the runtime dependency and packaged assets in sync across future changes.
- Wheel packaging for `rfmcp-skills` is now covered, but sdist-to-wheel rebuild behavior has not been explicitly proven in this epic.
- The built `rfmcp-skills` wheel was validated for contents, but install-and-run behavior from the built wheel itself was not explicitly exercised in this epic.
- External review capacity is a real delivery dependency:
  model timeouts and usage limits can interrupt the intended multi-review loop.

Significant discoveries:

- Host-level workflow documentation and pure-Python helper behavior must be distinguished explicitly; otherwise the product overclaims MCP activity that the helper itself does not perform.
- Publishing raw command templates as if they were executable commands creates avoidable operator confusion; example commands and rendered commands need separate surfaces.
- Benchmark evidence is most trustworthy when abort paths emit structured events too, not only success paths.
- A bounded live-state surface can still drift upward in tool count even if the architectural purpose stays narrow; that count should be reviewed explicitly as new context and inspection features are added.

### Key Insights and Learnings

- Determinism claims should be treated as proof obligations, not as architecture adjectives.
- Package-data and installation-path checks belong in the acceptance loop whenever a story introduces reusable workflow assets.
- Fail-closed provider and hint-pack behavior is worth the extra friction because it prevents silent guidance drift.
- Idempotence testing needs to cover semantic equivalents, not just exact string matches.
- Speculative scaffolding should be removed quickly when review proves it has no authoritative role; carrying dead helpers forward only increases drift risk.
- Critical review tools are most useful when the prompt bundle is narrow enough to finish reliably and concrete enough to drive executable checks.
