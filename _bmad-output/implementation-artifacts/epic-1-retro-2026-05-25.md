# Epic 1 Retrospective: Deliver the First Deterministic Validation Workflow

Date: 2026-05-25
Epic: 1
Status: complete

## Part 1: Epic Review

### Outcome Summary

Epic 1 finished with all five stories marked `done` in sprint tracking and with the shared verification suite passing.

Delivered in this epic:

- root `uv` workspace bootstrap baseline
- contributor-visible package scaffolds for core, MCP, CLI, skills, bundles, and first providers
- shared contract/model layer plus committed JSON Schemas
- first `rfmcp validate` CLI command with human and JSON output
- local policy defaults and file-first observability/event foundations

Verification evidence at close:

- `uv run --group dev python -m unittest discover -s tests -v`
- `uv run --group dev python scripts/verify_schema_sync.py`
- `python3 scripts/verify_workspace_structure.py`
- `uv run --group dev rfmcp validate README.md --json`

### What Went Well

- The epic sequencing worked: each story created a useful base for the next one instead of forcing rewrites.
- Shared verification scripts were added early enough to catch structural drift before Epic 2 work begins.
- Review-driven fixes improved the actual architecture, not just the story records:
  - bootstrap verifier root-path handling
  - workspace structure verifier root-path handling
  - workspace package source wiring for `uv`
  - installed CLI import path for bootstrap validation
- The package boundaries stayed aligned to the architecture rather than collapsing into a single ad hoc package.
- Contract-first work in Story 1.3 reduced ambiguity for the validate workflow in Story 1.4.

### What Went Poorly

- Multiple issues were variants of the same systems problem: repo-root assumptions leaked into scripts and tests.
- Workspace metadata requirements were under-specified at first:
  - `tool.uv.sources` had to be added only after inter-package references failed.
  - root dev dependencies needed explicit wiring for contract-layer scripts.
- The final code-review loop was partially degraded by tool constraints:
  - Codex review hit a usage-limit error during the final pass.
  - Claude CLI review attempts were inconsistent for larger prompts in this environment.
- Story-state bookkeeping lagged implementation at several points and had to be normalized afterward.

### Root Causes and Patterns

- The largest recurring defect class was environment coupling:
  scripts assumed a caller working directory, runtime package availability, or editable import path that was true during development but not guaranteed in actual execution.
- Review feedback was valuable when it forced execution through real invocation paths:
  `cd scripts && python3 ...`, `uv run ...`, and installed CLI entrypoints surfaced defects that unit-level reasoning alone missed.
- Story creation quality was strongest when the artifact called out exact architecture boundaries and file locations; implementation quality dropped whenever the runtime surface was inferred from memory instead of executed.

### Technical Debt and Residual Risks

- The local machine still does not meet the declared `uv` baseline, so the happy-path `rfmcp validate` command remains blocked by bootstrap enforcement until `uv 0.11.16` is installed.
- Package scaffolds are structurally correct but still skeletal:
  they do not yet include hook entry points, richer package metadata, or actual runtime implementations beyond the pieces needed for Epic 1.
- The validate workflow is intentionally minimal and static:
  it is a valid v1 slice, but behavioral verification remains deferred to later epics.
- Review automation itself is a process risk because external usage limits can interrupt the intended dual-review workflow.

### Key Lessons Learned

- Any script that may be run from more than one directory should anchor to repo-root or explicit arguments immediately.
- For workspace repos, testing only source imports is insufficient; `uv run` and installed entrypoints must be exercised before a story is considered closed.
- Contract-first implementation was the right choice. Once the payloads existed, CLI, schema, and observability work became materially easier to reason about.
- Verification utilities should live in package code when runtime features depend on them; script wrappers can delegate to package logic.

## Part 2: Next Epic Preparation

### Epic 2 Preview

Epic 2 is `Repair Failing Robot Tests with Live Context`. It depends directly on foundations completed in Epic 1:

- workspace/package boundaries
- shared contracts and schema discipline
- local policy defaults
- observability/event primitives
- the initial deterministic validation surface

### Ready for Epic 2

- Shared package structure exists and is verifiable.
- Shared error envelope, hint payload, hint pack, skill manifest, and validation result contracts are in place.
- Local policy defaults already express the expected baseline posture:
  attach disabled, loopback-only HTTP, explicit opt-in required.
- CLI and MCP event emitters exist for local benchmark and diagnostic capture.

### Gaps to Address Early in Epic 2

- Upgrade the local `uv` installation to the declared baseline so the validate command can succeed past bootstrap checks during normal development.
- Decide whether the bootstrap gate should remain first in every CLI workflow or whether some commands need a narrower compatibility check.
- Add provider/plugin metadata and hook surfaces before deeper repair hinting work begins, otherwise Epic 2 may accumulate more placeholder-only packages than useful extension seams.
- Keep testing focused on real invocation paths for MCP and CLI, not just direct Python imports.

### Action Items

- `Developer`: keep repo-root and invocation-path checks mandatory for all new scripts and CLI entrypoints.
- `Developer`: exercise future CLI and MCP features through `uv run` or equivalent installed paths before marking stories complete.
- `Developer`: upgrade the local `uv` toolchain to `0.11.16` before relying on the validate workflow for non-bootstrap debugging.
- `Architect/Developer`: define provider entry-point expectations explicitly at the start of Epic 2 so repair hinting does not drift into ad hoc plugin loading.

### Carry Forward

- Preserve the contract-first approach from Story 1.3.
- Preserve file-first, local-only policy and observability behavior.
- Continue treating review findings that appear only under real execution as higher-signal than static reasoning alone.
