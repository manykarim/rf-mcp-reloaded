### What Went Well

- The epic progressed in a clean dependency chain:
  grounding and scaffolding came first, then runnable generation, then refactor/regenerate, then shared skill packaging, then benchmark-backed proof.
- The implementation stayed aligned with the architecture boundary:
  generation and refactor behavior remained on deterministic CLI and skill surfaces instead of leaking into a larger MCP helper surface.
- Contract-first delivery paid off again:
  each user-facing workflow landed with typed payloads, schema exports, machine-usable JSON output, and focused regression coverage instead of relying on presenter behavior alone.
- Review-driven fixes improved the product materially rather than cosmetically:
  - generation learned to reject whitespace-only requests and report scaffold-injection failures structurally
  - refactor/regenerate learned rollback behavior, `.resource` support, `*** Tasks ***` support, and safer replacement failure shaping
  - benchmark proof learned to distinguish substantive determinism from temp-path noise and to report less misleading metrics
- Epic 3 now has real local proof surfaces for authoring behavior:
  runnable generation, runnable refactor, packaged skill assets, and a benchmark pack all exist as executable evidence instead of planning-only claims, even though CI enforcement of that proof is still future Epic 4 work.

### Challenges and Growth Areas

- The dominant defect class remained runtime-surface coupling:
  temp paths, installed package contents, wheel-build inclusion, and real `robot` execution exposed issues that source-only tests would not have caught.
- Benchmark semantics were not correct on the first pass:
  determinism checks, elapsed timing, and correction-burden accounting needed extra review pressure before the proof report became trustworthy.
- External review tooling remained valuable but uneven:
  Copilot Sonnet surfaced real benchmark issues, MiniMax produced useful PASS/fail checks, and some larger review attempts still stalled or timed out before returning a final verdict.
- Epic 3 completed the authoring and proof side of the plan, but it did not close the environment-policy gap inherited from earlier epics:
  the local `uv` baseline mismatch and the bootstrap-vs-workflow gating question still remain.

### Root Causes and Patterns

- Path-sensitive output is still a recurring source of false confidence:
  diffs, execution transcripts, and follow-up text can look non-deterministic even when workflow behavior is stable, so comparison logic needs to normalize carefully.
- “Machine-usable” is not the same as “release-credible”:
  a JSON report can still mislead if fields are hardcoded, semantically mixed, or only loosely asserted in tests.
- The highest-signal tests remain end-to-end checks that cross package, CLI, validation, and runtime seams together.
- Review bundles worked best when they were artifact-scoped:
  the narrower story-specific bundles produced actionable findings, while larger cross-cutting bundles were more likely to degrade into tool-runtime issues.

### Technical Debt, Residual Risks, and Significant Discoveries

- The local environment is still below the declared `uv` baseline, so bootstrap-sensitive workflow claims continue to live under a known mismatch.
- The bootstrap-vs-workflow gating rule is still unresolved, which risks inconsistent compatibility behavior as more host-facing entrypoints land in Epic 4.
- The benchmark pack now provides trustworthy small-scale proof, but CI enforcement of that proof is still future work in Epic 4.
- Canonical skills and fallback docs now exist, but first-class host rendering and installation claims are still unproven until the supported-host stories land.
- External review capacity remains a delivery dependency:
  tool timeouts did not stop Epic 3, but they did degrade at least one Story 3.4 review pass and still affect how confidently a wide bundle can be audited in one shot.

Significant discoveries:

- Determinism checks need explicit canonicalization boundaries:
  normalize temp-path and filename noise, but do not normalize away substantive diff or execution payload differences.
- Benchmark fields become product surface area quickly once they are machine-readable:
  even a small semantic mistake in summary fields or correction metrics weakens release evidence.
- Host-agnostic skill packaging works best when canonical definitions, assets, and fallback commands are treated as shared source of truth and renderers are kept downstream.
- Generation and refactor workflows are tractable to maintain when they reuse the same validation, diagnostics, hinting, and proof structures instead of growing bespoke result types.

### Key Insights and Learnings

- Runnable proof should be designed as part of the workflow, not added as a final verification garnish.
- Benchmark evidence needs the same review rigor as executable code because downstream release claims will lean on it.
- Wheel-build and package-data checks belong in story verification whenever a feature ships reusable assets or docs.
- Narrow, concrete review prompts produced better findings than broad “review everything” bundles.
- The project is ready to scale host onboarding only because Epic 3 kept the authoring workflows deterministic and shared rather than host-specific.
